# 正式前端页面结构规划

更新日期：2026-03-23

## 1. 目标

把当前内置 [workbench.html](/Users/zhangchao/Desktop/chaxunxitong/backend/app/ui/workbench.html) 升级为可长期使用的正式前端。

当前 `/workbench` 已经验证过的真实能力有：

- 成本试算
- 方案导入
- Agent 生成
- 方案列表
- 方案详情
- 方案重算
- 导入预检报告
- PPT 下载入口

正式前端的目标不是“换个皮肤”，而是把这些能力拆成稳定、清晰、可维护的后台信息架构。

## 2. 设计原则

1. **任务优先**
   页面按真实工作流组织，不按数据库表生硬拆页。

2. **列表 + 详情 + 操作台**
   中后台高频模式尽量统一，减少学习成本。

3. **先做可用，再做高级**
   先把商品、成本、方案、匹配、Agent 五条主链路做完整，再补看板和高级配置。

4. **尽量少跳页**
   高频操作优先用抽屉、侧栏、页内分区完成，避免反复返回列表。

5. **试用页向正式页平滑迁移**
   当前 `/workbench` 里的模块应该直接映射到正式前端页面，不要完全推翻。

## 3. 推荐的正式前端总结构

推荐采用左侧导航 + 顶部上下文区 + 主内容区。

### 左侧一级导航

1. 仪表盘
2. 商品中心
3. 成本中心
4. 匹配中心
5. 方案中心
6. Agent 工作台
7. 导入与报告
8. 系统设置

### 顶部区域

- 当前页面标题
- 面包屑
- 全局搜索
- 当前用户
- 快捷入口

建议保留一个全局快捷动作区：

- 新建商品
- 上传方案
- 打开 Agent
- 查看待确认匹配

## 4. 页面信息架构

## 4.1 仪表盘

路由建议：

- `/dashboard`

页面目标：

- 给管理者和操作人员一个总览入口

页面内容：

- 核心指标卡片
  - 商品数
  - 成本记录数
  - 待确认匹配数
  - 方案数
  - Agent 生成数
- 最近导入记录
- 最近方案
- 待处理队列
  - 低置信度匹配
  - 失败导入
  - 待补成本商品

这一页先做轻，不要一开始做复杂 BI。

## 4.2 商品中心

路由建议：

- `/products`
- `/products/brands`
- `/products/:productId`

页面结构：

### 商品列表页

- 筛选条件
  - 关键词
  - 品牌
  - 分类
  - 状态
  - 是否缺条码
- 列表字段
  - 商品名称
  - 品牌
  - 条码
  - 分类
  - 规格
  - 状态
  - 最近成本
  - 更新时间
- 行操作
  - 查看详情
  - 编辑
  - 归档/启用

### 商品详情页

建议做成“详情页 + 标签页”：

- 基本信息
- 别名
- 图片
- 成本历史
- 匹配记录

### 品牌管理页

- 品牌列表
- 新建品牌
- 编辑品牌
- 品牌状态管理

## 4.3 成本中心

路由建议：

- `/costs`
- `/costs/imports`
- `/costs/products/:productId`

页面结构：

### 成本总览页

- 筛选条件
  - 商品
  - 品牌
  - 供应商
  - 时间范围
- 列表字段
  - 商品
  - 当前有效成本
  - 生效时间
  - 供应商
  - 来源

### 商品成本历史页

- 时间轴 / 表格
- 支持查看指定商品所有成本变更
- 支持新增一条成本
- 支持人工覆盖/修正

### 成本导入记录页

- 导入批次列表
- 成功数 / 失败数
- 错误报告下载

## 4.4 匹配中心

路由建议：

- `/match/review`
- `/match/mappings`
- `/match/logs`
- `/match/feedback`

页面结构：

### 待确认队列

这是匹配中心最重要的一页。

- 列表字段
  - 原始输入
  - 当前推荐商品
  - 置信度
  - 匹配方法
  - 时间
- 操作
  - 确认当前推荐
  - 改选其他候选
  - 标记无匹配

### 映射管理页

- 手动映射列表
- 自动映射列表
- 启用 / 停用
- 查看来源和命中次数

### 匹配日志页

- 查询词
- 最终商品
- 置信度
- 方法
- 时间

### 反馈页

- 错误映射反馈
- 处理结果回看

## 4.5 方案中心

路由建议：

- `/solutions`
- `/solutions/import`
- `/solutions/:solutionId`

页面结构：

### 方案列表页

- 筛选条件
  - 关键词
  - 场景
  - 行业
  - 状态
- 列表字段
  - 方案名称
  - 场景
  - 行业
  - 商品条数
  - 总成本
  - 待确认条数
  - 创建时间

### 方案导入页

- 上传 `.xlsx` / `.csv`
- 填元信息
  - 场景
  - 行业
  - 创建人
- 展示解析状态

### 方案详情页

这页是正式前端的重点页之一。

推荐拆成四块：

1. 基本信息区
2. 成本摘要区
3. 明细表格区
4. 操作侧栏

明细表格区建议字段：

- 原始名称
- 匹配商品
- 数量
- 单位
- 单价
- 小计
- 匹配方式
- 是否待确认

需要支持的操作：

- 改数量
- 改单位
- 改匹配商品
- 触发重算
- 标记确认
- 导出 / 触发 PPT

路由建议：

## 5. 推荐的第一版正式前端范围

第一版不要把所有页一次做完。

建议先做：

1. 商品中心
2. 成本中心
3. 匹配待确认队列
4. 方案列表 + 详情
5. Agent 工作台

仪表盘和设置页可以先简化。

## 6. 从当前 `/workbench` 到正式前端的映射关系

当前试用模块与正式页面的对应关系：

- 成本试算 -> `/agent` 或 `/costs` 下的快速试算面板
- 方案导入 -> `/solutions/import`
- Agent 生成 -> `/agent`
- 方案列表 -> `/solutions`
- 方案详情 / 重算 -> `/solutions/:solutionId`
- 导入预检报告 -> `/imports/precheck`

所以正式前端并不是另起炉灶，而是把当前试用能力拆到更清晰的页面中。

## 7. 推荐的开发顺序

### Phase 1

- 搭正式前端工程壳子
- 做全局布局
- 接商品列表/详情
- 接方案列表/详情

### Phase 2

- 接成本中心
- 接匹配待确认队列
- 接导入预检报告

### Phase 3

- 接 Agent 工作台
- 接 PPT 下载与状态
- 做统一异常提示和权限占位

## 8. 页面层级建议

建议尽量控制在三级以内：

- 一级：业务模块
- 二级：列表 / 工作台 / 管理页
- 三级：详情页

不要把前端路由拆得过碎，否则中后台使用体验会很差。

## 9. 建议给设计/前端的结论

你可以直接把下面这段发给设计或新前端团队：

> 当前已有一个内置 `/workbench` 验证主链路，但它只是试用工作台。正式前端请基于“商品中心、成本中心、匹配中心、方案中心、Agent 工作台、导入与报告”这 6 个主模块搭建后台信息架构，优先保证方案详情编辑、匹配确认、Agent 生成这 3 个核心工作流。
